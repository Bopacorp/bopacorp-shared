## Description 

Develop the software system selected and designed in our previous module of Software Engineering I, using a development methodology and software tools to achieve a final product appropriately tested and versioned with a clean code. The system must be developed in teams composed by 3 to 5 members. All the reports, documents, communications, etc. must be done in English; although, if needed, the pieces of evidence of the communications with your client can contain a translation to Spanish. 

The deployment to production will be monitored and this deployment is only accepted after the first sprint. At the end of each Sprint, you are expected to provide: 

- An acceptance form signed by the customer. 

- Evidence of sprint review. 

- The software system deployed into a production environment. 

## Reporting Communications 

The name of your repository must be the same than your team (T1, T2, …, Tn). This repository must contain a folder named _Communications_ . This folder should keep all the pieces of evidence of the communications between you and your client. Some examples of pieces of evidence include: (i) instant messaging (e.g., WhatsApp, Signal); (ii) emails; (iii) voice messages or recordings; (iv) videos; (vii) commitment/acceptance letter. 

Create a document (table of content) to index all the piece of evidence within the communications folder. Each entrance in the index must specify (i) the name of the file, (ii) a brief description of its content; (iii) the file type; (iv) the participants; (v) date and time, among others. The document must be constantly updated until the deployment of the final version of the software system. 

## Section A: Project Report and Implementation 

Each team must write a project report, in English, that adheres to the template and guidelines specified in the reference “Writing guidelines and templates” [3] from ETS but using ESPOL logo [6]. The report must be a **self-contained document** with pieces of evidence about the project context, development process, testing levels, deployment, etc.

## Section B: Project Presentation 

The team must perform a demonstration (in English) of your software system. You must record the demonstration and upload it to your repository. The demonstration is expected to last 10 minutes and include equal participation of all team members.  The team must show the software components running and demonstrate their compliance with the functional and non-functional requirements. The team must also present the results of the evaluation of the system, covering aspects of your system design and other points covered in the project report, including a review of your test plans. The team is also encouraged to upload the presentation file used during your demonstration. 

Some aspects that your presentation must include are: 

1. a brief introduction of the system (who the client is, system scope, what the system does). 

2. The number of user stories the system has. 

3. The number of sprints needed to develop the system. 

4. The architecture (deployment and component diagrams). 

5. How users interact with the system by performing test cases (acceptance testing based on the user stories and acceptance criteria). Be sure that one user per role appears in the demonstration. 

6. Other information you consider important. 

The presentation must be prepared using ESPOL official templates [6], attached to this specification. 

## Deliverables 

1. A self-contained report of the project (in pdf) written in English that includes (the uploaded file must be named as T<team number><ProjectName>.pdf): 

   - a. a compilation of relevant documents, charts, reports, and any other information that supports your work in reference to the elements assessed in the marking scheme. 

   - b. The repository’s URL with the source code, resources, communication, and SCRUM evidence. 

   - c. Access to any online tool to support any of the elements assessed in the marking scheme. 

   - d. An acceptance form signed by the customer. 

   - e. The recorded video of your project presentation.

2. The software system deployed into a production environment. 

3. Each student must submit an evaluation of other team members using the co-evaluation scheme in the **individual assignment** “Project Co-evaluation”, available in Aula Virtual.
